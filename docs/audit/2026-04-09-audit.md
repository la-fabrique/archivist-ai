# Audit critique — `packages/referentiel/` v0

**Date** : 2026-04-09
**Cible** : indep SASU solo, 0-1 salarié | **Axes** : fond métier + utilisabilité LLM
**Périmètre validé** : URSSAF/DSN, assurances pro, PI/licences
**Note** : le référentiel est lu et appliqué par un LLM (bot de classement), pas directement par un humain

---

## 1. MANQUES CRITIQUES — Documents absents du périmètre

### 1.1 Aucune trace de l'URSSAF / cotisations sociales

Le dirigeant SASU est assimilé salarié. Il reçoit chaque mois/trimestre :

- **Bordereaux URSSAF** (déclarations de cotisations)
- **DSN** (Déclaration Sociale Nominative) — le document social central depuis 2017
- **Attestations de vigilance** — exigées par tout client pour un contrat > 5 000 €
- **Relevés de situation de compte URSSAF**

**Aucun de ces documents n'a de place dans l'arborescence.** `Mon social/` ne contient que fiches de paie, DPAE et contrats de travail.

**Recommandation** : ajouter dans `Mon social/` au minimum `Mes déclarations sociales/` (DSN) et `Mes attestations URSSAF/`.

### 1.2 Assurances : mentionnées dans les durées de conservation, introuvables dans l'arborescence

`regles-archivage.md` donne une durée de conservation pour les assurances (2 ans après fin de contrat). Mais **aucun dossier** n'existe pour les stocker. L'indep SASU a typiquement :

- RC Pro (souvent obligatoire)
- Mutuelle/prévoyance (obligatoire avec salarié)
- Assurance locaux/matériel

**Recommandation** : créer `Mes assurances/` avec sous-dossiers par type, ou a minima un sous-dossier dans `Mon juridique/`.

### 1.3 Propriété intellectuelle / licences : inexistant

- Dépôts de marque INPI
- Licences logicielles (SaaS, perpétuelles)
- Cessions de droits d'auteur
- Contrats de licence

Rien de prévu. Pour un indep dans le numérique ou le créatif, c'est pourtant central.

**Recommandation** : sous-dossier dans `Mon juridique/` type `Mes marques et licences/`.

### ~~1.4 Pas de gestion de la relation comptable~~

> **HORS PÉRIMÈTRE v0.** La synchro comptable (Dext, Pennylane, workflow de transmission) n'est pas traitée dans cette version.

## 2. INCOHÉRENCES ET ERREURS MÉTIER

### 2.1 `Ma banque et caisse/` est squelettique et mal nommé

Un seul sous-dossier (`Mes relevés bancaires/`). Problèmes :

- **Où vont les RIB ?** Demandés en permanence (clients, fournisseurs, administration). Pas de place.
- **"Caisse"** dans le nom mais rien pour la caisse. Pour un indep SASU sans activité cash, c'est du bruit.
- **Plusieurs comptes bancaires** : un indep SASU a souvent un compte pro + un compte épargne pro. La structure plate ne distingue rien.

**Préconisation** :

```
Ma banque/                              ← renommer (supprimer "et caisse")
├── Mes relevés bancaires/
│   └── [Nom banque]/                   ← un sous-dossier par établissement
│       └── [AAAA-MM]/
└── Mes RIB/                            ← nouveau
```

Si un profil "commerçant avec caisse" est ajouté plus tard, on ajoutera `Ma caisse/` via le système de modules (cf. §4).

### 2.2 `Mon juridique/` est trop maigre pour une SASU

Trois sous-dossiers (Statuts, K-bis, PV d'AG). Il manque :

- **Registre des décisions de l'associé unique** — obligatoire en SASU, distinct des PV d'AG
- **Registre des mouvements de titres** — obligatoire
- **Baux commerciaux / contrats de domiciliation**
- **CGV / mentions légales** — documents juridiques fondamentaux pour tout indep

**Préconisation** :

```
Mon juridique/
├── Mes statuts/                        ← existant
├── Mes K-bis/                          ← existant
├── Mes PV et décisions/               ← renommer "Mes PV d'assemblée/"
│                                         inclut PV AG + décisions associé unique
├── Mes registres/                      ← nouveau
│   ├── Registre des mouvements de titres/
│   └── Registre des décisions/
├── Mes baux et domiciliation/          ← nouveau
├── Mes CGV et mentions légales/        ← nouveau
└── Mes marques et licences/            ← nouveau (cf. §1.3, module optionnel)
```

### 2.3 Confusion devis/offre dans `Mes ventes/`

Le référentiel crée trois dossiers de modèles distincts (`Mes modèles de contrat/`, `Mes modèles de devis/`, `Mes modèles d'offre/`) mais pour un indep solo, la distinction devis/offre est souvent inexistante (un devis EST l'offre).

**Préconisation** — consolider :

```
Mes ventes/
├── Mes factures clients/
│   └── [AAAA-MM]/
├── Mes modèles/                        ← un seul dossier de modèles
│   ├── Contrats/
│   └── Devis et offres/               ← fusionné
└── Mes clients/
    └── [Nom du client]/
        ├── Contrats/
        ├── Devis/                      ← inclut offres
        └── ...
```

## 3. INCOHÉRENCES DE NOMMAGE

Le référentiel est destiné à être lu par un LLM. Les règles de nommage doivent être sans ambiguïté. Or `regles-nommage.md` contient des contradictions internes et des divergences avec les fichiers `classement/*.md`.

### 3.1 Le socle commun est contredit par ses propres sections spécifiques

Le socle commun définit : *"Majuscule initiale sur chaque segment"* et *"Tiret `-` dans un segment, underscore `_` entre les segments"*.

Mais les sections spécifiques de `regles-nommage.md` violent ces règles :

| Section | Exemple donné | Violation |
|---------|---------------|-----------|
| `Mon social/` | `2026-03_fiche-paie_dupont-jean.pdf` | Tout en minuscules (devrait être `Fiche-paie_Dupont-Jean`) |
| `Ma fiscalité/` | `2026-01_declaration_tva.pdf` | Minuscules + underscore intra-segment (`declaration_tva` → devrait être `Declaration-TVA`) |
| `Ma fiscalité/` | `2026_avis_cfe.pdf` | Idem (`avis_cfe` → `Avis-CFE`) |
| `Ma banque/` | `2026-03_releve_banque-populaire.pdf` | Minuscules (`releve_banque-populaire` → `Releve_Banque-Populaire`) |
| `Mon juridique/` | `2025_statuts_sas-monentreprise.pdf` | Minuscules (`statuts` → `Statuts`) |

**Fichier** : `regles-nommage.md`, sections par dossier racine.

### 3.2 Divergences entre `regles-nommage.md` et les fichiers `classement/*.md`

Les fichiers `classement/*.md` appliquent généralement le socle commun correctement, mais contredisent les exemples de `regles-nommage.md` :

| Divergence | `regles-nommage.md` | Fichier classement | Problème |
|------------|---------------------|--------------------|----------|
| Date juridique | `AAAA` (année seule) | `mon_juridique.md` : `AAAA-MM` | Contradiction directe sur la granularité de date |
| Prénom salarié | `dupont-jean` (inclus) | `mon_social.md` : `Dupont` (absent) | Segment `[Nom salarié]` incohérent |
| Numéro relevé | absent du format | `ma_banque_et_caisse.md` : `_[Numero]` en 4e segment | Format à 3 segments vs 4 segments |
| Objet modèle contrat | `maintenance_annuelle` (underscore) | Socle commun : tiret dans un segment | `mes_ventes.md` ET `regles-nommage.md` partagent cette violation |

**Fichiers** : `regles-nommage.md` vs `mon_juridique.md`, `mon_social.md`, `ma_banque_et_caisse.md`, `mes_ventes.md`.

### 3.3 Accent interdit mais présent dans un exemple

`mon_social.md` donne l'exemple `2026-06_Avenant_Dupont_télétravail.pdf`. Le socle commun interdit explicitement les accents (*"Accents : Jamais"*, *"`fiscalite` pas `fiscalité`"*).

**Fichier** : `mon_social.md`, section Avenants.

## 4. MODULARITÉ — Proposition détaillée

### 4.1 Principe : un noyau obligatoire + des modules optionnels

Le référentiel se compose de :

- **Noyau (`core`)** : dossiers et règles présents pour tout profil indep/SASU
- **Modules** : extensions activables selon le contexte métier

Chaque fichier `classement/*.md` porte un frontmatter YAML qui déclare son appartenance :

```yaml
---
id: mon_social
module: core            # ou le nom du module
profiles: [sasu-solo, sasu-employeur]
required: true          # false = le dossier peut être vide/absent
---
```

### 4.2 Définition du noyau (`core`)

Dossiers toujours présents, quel que soit le profil :

| Dossier | Justification |
|---------|---------------|
| `Mes ventes/` | Tout indep facture |
| `Mes achats/` | Tout indep a des fournisseurs |
| `Mon juridique/` | Statuts, K-bis, CGV — obligatoire |
| `Ma fiscalité/` | TVA, liasse, avis — obligatoire |
| `Ma banque/` | Relevés, RIB — obligatoire |
| `Archives/` | Cycle d'archivage — obligatoire |

### 4.3 Modules optionnels

#### Module `employeur` — Activé si ≥ 1 salarié

Ajoute/enrichit :

| Dossier | Contenu |
|---------|---------|
| `Mon social/` (nouveau dossier racine) | Fiches de paie, DPAE, contrats de travail, déclarations sociales |
| `Mon social/Mes salariés/` | Sous-dossier par salarié |
| `Mon social/Mes déclarations sociales/` | DSN mensuelle |

Sans ce module, `Mon social/` n'est pas déployé.

#### Module `dirigeant-assimile-salarie` — Activé si SASU

Ajoute :

| Dossier | Contenu |
|---------|---------|
| `Mon social/Mes fiches de paie/` | Fiches de paie du dirigeant (même si pas de salarié) |
| `Mon social/Mes attestations URSSAF/` | Attestations de vigilance, bordereaux |

Ce module active `Mon social/` même sans salarié.

#### Module `assurances` — Activé si RC Pro, mutuelle, ou assurance locaux

Ajoute :

| Dossier | Contenu |
|---------|---------|
| `Mes assurances/` (nouveau dossier racine) | RC Pro, mutuelle/prévoyance, assurance locaux |

#### Module `propriete-intellectuelle` — Activé si marques, brevets, ou licences

Ajoute :

| Dossier | Contenu |
|---------|---------|
| `Mon juridique/Mes marques et licences/` | Dépôts INPI, licences logicielles, cessions de droits |

#### Module `brevets` — Activé si brevets déposés

Ajoute :

| Dossier | Contenu |
|---------|---------|
| `Mon juridique/Mes brevets/` | Brevets déposés, certificats, correspondance INPI |

### 4.4 Profils prédéfinis (combinaisons de modules)

| Profil | Modules activés |
|--------|----------------|
| **Indep solo micro** | `core` |
| **SASU solo sans salarié** | `core` + `dirigeant-assimile-salarie` + `assurances` |
| **SASU solo + 1 salarié** | `core` + `dirigeant-assimile-salarie` + `employeur` + `assurances` |
| **Indep créatif/tech** | `core` + `propriete-intellectuelle` |
| **SASU créatif/tech + salarié** | `core` + `dirigeant-assimile-salarie` + `employeur` + `assurances` + `propriete-intellectuelle` |

### 4.5 Implémentation technique

Le CLI (`referentiel-cli`) lit les frontmatter de chaque fichier `classement/*.md` et :

1. Demande le profil à l'utilisateur (ou le lit depuis un fichier de config)
2. Résout les modules activés
3. Ne déploie que les dossiers correspondants
4. Ignore les fichiers dont le `module` n'est pas activé

## 8. VERDICT

Le squelette est bon (classement par fonction métier, c'est le bon choix), la documentation est soignée. Deux chantiers prioritaires :

1. **Corriger les incohérences de nommage** — le LLM ne peut pas appliquer des règles qui se contredisent entre `regles-nommage.md` et les fichiers `classement/*.md`
2. **Combler les trous métier** — URSSAF/DSN, assurances, registres obligatoires SASU

La modularité par profils/modules permettra d'ajouter des déclinaisons métier sans alourdir le noyau.
